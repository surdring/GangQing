## 执行摘要

### 核心决策

**不直接依赖 CopilotKit，借鉴其优秀设计模式，自研符合 GangQing 工业场景强制约束的 AI Copilot 核心组件。**

### 关键理由

1. **安全合规优先**：CopilotKit 无法满足工业场景的只读默认、审计不可篡改、Kill Switch、OT 写入门禁等强制约束
2. **私有化部署要求**：模型与数据必须在内网，不可出厂
3. **证据链强制**：所有数值结论必须可追溯到数据源（CopilotKit 无内置支持）
4. **多租户隔离**：tenantId/projectId 强制过滤（CopilotKit 无内置支持）
5. **技术债务控制**：避免 fork 开源项目并持续跟进上游更新的维护成本

### 预期收益

1. **开发效率提升 40%**：通过装饰器自动注册、依赖注入、统一封装减少样板代码
2. **安全风险降低 80%**：自动化 RBAC/脱敏/审计，减少人为遗漏
3. **可维护性提升 50%**：统一架构模式，降低新人上手成本
4. **审计覆盖率 100%**：所有操作自动记录，满足合规要求

## 1. 整体架构设计

### 1.1 分层架构

```
┌─────────────────────────────────────────────────────────────┐ │ 前端层（React + TypeScript + Zod） │ │ - 上下文感知组件（借鉴 CopilotKit） │ │ - SSE 客户端状态机（自动重连/错误处理/取消传播） │ │ - 证据链可视化（GangQing 独有） │ └─────────────────────────────────────────────────────────────┘ ↓ SSE/WebSocket ┌─────────────────────────────────────────────────────────────┐ │ API 网关层（FastAPI + Pydantic） │ │ - 请求上下文自动注入（借鉴 CopilotKit） │ │ - SSE Envelope 统一封装（增强版） │ │ - 统一错误模型（GangQing 强制） │ └─────────────────────────────────────────────────────────────┘ ↓ ┌─────────────────────────────────────────────────────────────┐ │ 编排层（Orchestration Engine） │ │ - 意图识别与策略路由（GangQing 独有） │ │ - 工具链注册与调用（借鉴 CopilotKit） │ │ - Evidence 引擎（GangQing 独有） │ │ - 只读默认门禁（GangQing 独有） │ └─────────────────────────────────────────────────────────────┘ ↓ ┌─────────────────────────────────────────────────────────────┐ │ 工具与适配层（Tools & Adapters） │ │ - 装饰器自动注册（借鉴 CopilotKit） │ │ - RBAC + 脱敏 + 审计（GangQing 强制） │ │ - 超时重试 + 降级（GangQing 强制） │ └─────────────────────────────────────────────────────────────┘ ↓ ┌─────────────────────────────────────────────────────────────┐ │ 数据层 + 模型推理层 │ │ - Postgres（多租户隔离 + 审计 append-only） │ │ - llama.cpp（私有化部署 + 并发控制） │ │ - Elasticsearch（审计检索增强） │ └─────────────────────────────────────────────────────────────┘
```


### 1.2 核心设计原则

#### 1.2.1 安全第一
- **只读默认**：任何写操作必须经过草案→审批→执行→回滚全链路
- **审计不可篡改**：所有操作 append-only 记录，可按 requestId 追溯
- **Kill Switch**：一键熔断所有写操作能力
- **OT 写入门禁**：专用通道 + 二次确认，禁止 AI 直连 DCS/PLC

#### 1.2.2 证据驱动
- **强制可追溯**：所有数值结论必须绑定数据源、时间范围、口径版本
- **降级规则**：证据缺失/不可验证/冲突时必须降级并输出 warning
- **增量更新**：工具调用后实时输出 evidence.update 事件

#### 1.2.3 契约强制
- **单一事实源**：前端 Zod、后端 Pydantic，运行时校验
- **向后兼容**：对外契约只增不改，破坏性变更需版本化
- **契约测试**：自动化测试覆盖所有对外接口

#### 1.2.4 多租户隔离
- **强制过滤**：所有数据访问默认叠加 tenantId/projectId 过滤
- **跨域拒绝**：检测到跨租户/项目访问立即拒绝并审计
- **字段脱敏**：按角色配置字段级脱敏策略

#### 1.2.5 可观测性
- **requestId 贯穿**：HTTP → 编排 → 工具 → 审计 → 对外响应
- **结构化日志**：JSON 格式，包含 requestId/tenantId/projectId
- **Traces/Metrics**：OpenTelemetry 集成，P50/P95/P99 可观测

## 2. 核心组件设计

### 2.1 前端：上下文感知与流式渲染

#### 2.1.1 设计目标

**借鉴 CopilotKit 的 `useCopilotChat`，封装 SSE 复杂性，提供声明式 React Hook 接口。**

#### 2.1.2 核心能力

| 能力 | 说明 | 借鉴来源 |
|------|------|---------|
| **SSE 连接管理** | 自动连接/断开/重连（指数退避） | CopilotKit |
| **事件解析** | 统一解析 SSE Envelope，按 type 分发 | CopilotKit + 增强 |
| **状态管理** | idle/connecting/streaming/error/done | CopilotKit |
| **错误处理** | 解析结构化错误（ErrorResponse） | GangQing 独有 |
| **取消传播** | 客户端断开 → 服务端停止推理/工具调用 | CopilotKit |
| **Zod 校验** | 运行时校验 SSE 事件契约 | GangQing 独有 |
| **证据链管理** | 增量更新 Evidence 列表 | GangQing 独有 |

#### 2.1.3 关键设计决策

**决策 1：为什么不直接使用 CopilotKit 的 `useCopilotChat`？**

| 维度 | CopilotKit | GangQing 需求 | 差距 |
|------|-----------|--------------|------|
| 事件模型 | 简洁（type + payload） | 严格（type + envelope + payload） | 需增加 requestId/tenantId/sequence |
| 错误处理 | 客户端边界 | 结构化错误（code/message/retryable） | 需解析 ErrorResponse |
| 证据链 | 无 | 增量更新（evidence.update） | 需自研 Evidence 管理 |
| 契约校验 | 无 | Zod 运行时校验 | 需自研 schema 校验 |

**决策 2：如何平衡复用与定制？**

- **复用**：SSE 连接管理、重连策略、状态机模式
- **定制**：事件解析、错误处理、证据链管理、契约校验

#### 2.1.4 实现策略

**阶段 1：最小可用版本（MVP）**
- 实现基础 SSE 连接与事件解析
- 支持 message.delta 流式渲染
- 支持 error/final 事件处理

**阶段 2：增强版本**
- 增加自动重连（指数退避）
- 增加取消传播
- 增加 Zod 运行时校验

**阶段 3：完整版本**
- 增加 evidence.update 增量更新
- 增加 progress/tool.call/tool.result 事件支持
- 增加性能优化（事件批处理、虚拟滚动）

#### 2.1.5 Context Panel（证据链可视化）

**设计目标**：直观展示数据来源与计算过程，支持降级态表达。

**核心能力**：
- **Trust Pill（数值胶囊）**：数值结论以胶囊形式展示，悬停显示数据源
- **Evidence 展开**：点击胶囊展开完整证据链（source/time_range/lineage）
- **降级态表达**：缺失/不可验证/冲突时显示虚线边框并提示
- **可追溯性**：支持跳转到原始数据源（如 ERP 报表、MES 工单）

**关键设计决策**：

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 展示位置 | 右侧固定面板 | 不遮挡对话区，便于对比 |
| 交互方式 | 悬停预览 + 点击展开 | 渐进式信息披露，降低认知负担 |
| 降级表达 | 虚线边框 + 图标 + 提示文案 | 视觉差异化，明确不确定性 |
| 性能优化 | 虚拟滚动 + 懒加载 | 支持大量 Evidence（100+ 条） |

### 2.2 后端：请求上下文自动注入

#### 2.2.1 设计目标

**借鉴 CopilotKit 的上下文管理，使用依赖注入实现自动提取与强制校验。**

#### 2.2.2 核心能力

| 能力 | 说明 | 强制约束 |
|------|------|---------|
| **自动提取** | 从请求头提取 requestId/tenantId/projectId | 缺失返回 AUTH_ERROR |
| **强制校验** | tenantId/projectId 必须存在且合法 | 不可绕过 |
| **依赖注入** | FastAPI Depends 自动注入到路由函数 | 减少手动传参 |
| **全链路贯穿** | HTTP → 编排 → 工具 → 审计 | requestId 可追溯 |

#### 2.2.3 关键设计决策

**决策 1：为什么使用依赖注入而非中间件？**

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| 中间件 | 全局生效，无需每个路由声明 | 难以测试，隐式依赖 | ❌ |
| 依赖注入 | 显式声明，易于测试，类型安全 | 每个路由需声明 | ✅ |

**决策 2：如何处理 requestId 未传入的情况？**

| 方案 | 说明 | 选择 |
|------|------|------|
| 拒绝请求 | 返回 AUTH_ERROR | ❌ 过于严格 |
| 自动生成 | 服务端生成并回传 | ✅ 兼容性好 |

**决策 3：如何处理 tenantId/projectId 缺失？**

| 方案 | 说明 | 选择 |
|------|------|------|
| 使用默认值 | 降级到默认租户/项目 | ❌ 安全风险 |
| 拒绝请求 | 返回 AUTH_ERROR | ✅ 强制隔离 |

#### 2.2.4 实现策略

**阶段 1：基础版本**
- 实现 RequestContext 数据类
- 实现 from_request 提取逻辑
- 实现 FastAPI Depends 注入

**阶段 2：增强版本**
- 增加 JWT 解析（提取 userId/role）
- 增加 tenantId/projectId 合法性校验（查数据库）
- 增加审计记录（上下文提取失败）

**阶段 3：完整版本**
- 增加上下文缓存（减少重复校验）
- 增加上下文传播（跨服务调用）
- 增加上下文监控（Prometheus metrics）

### 2.3 后端：工具装饰器自动注册

#### 2.3.1 设计目标

**借鉴 CopilotKit 的 `useCopilotAction`，使用装饰器简化工具注册，自动化 RBAC/脱敏/审计。**

#### 2.3.2 核心能力

| 能力 | 说明 | 自动化程度 |
|------|------|-----------|
| **声明式注册** | 装饰器声明工具元信息 | 100% 自动 |
| **RBAC 检查** | 基于 capabilities 自动检查权限 | 100% 自动 |
| **数据域过滤** | 基于 ctx.tenant_id/project_id 自动过滤 | 100% 自动 |
| **参数校验** | Pydantic 自动校验工具参数 | 100% 自动 |
| **脱敏输出** | 基于 masking_policy 自动脱敏 | 100% 自动 |
| **审计记录** | 自动记录 tool.call/tool.result | 100% 自动 |
| **Evidence 生成** | 自动生成 Citation/Lineage/ToolCallTrace | 100% 自动 |
| **超时重试** | 自动超时控制与重试 | 100% 自动 |

#### 2.3.3 关键设计决策

**决策 1：为什么使用装饰器而非基类继承？**

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| 基类继承 | 强制实现接口，类型安全 | 样板代码多，不够灵活 | ❌ |
| 装饰器 | 声明式，简洁，灵活 | 运行时检查，类型提示弱 | ✅ |

**决策 2：如何平衡自动化与灵活性？**

| 场景 | 自动化策略 | 灵活性保障 |
|------|-----------|-----------|
| 标准工具 | 100% 自动化（RBAC/脱敏/审计） | 通过装饰器参数配置 |
| 特殊工具 | 提供钩子函数（before/after） | 允许自定义逻辑 |
| 复杂工具 | 提供低级 API（手动控制） | 不强制使用装饰器 |

**决策 3：如何处理工具调用失败？**

| 失败类型 | 处理策略 | 审计记录 |
|---------|---------|---------|
| 参数校验失败 | 返回 VALIDATION_ERROR | tool.call + error |
| RBAC 拒绝 | 返回 FORBIDDEN | tool.call + error |
| 超时 | 重试 3 次，返回 UPSTREAM_TIMEOUT | tool.call + tool.result(failure) |
| 其他异常 | 返回 INTERNAL_ERROR | tool.call + tool.result(failure) |

#### 2.3.4 实现策略

**阶段 1：基础版本**
- 实现装饰器基础框架
- 实现自动注册到工具注册表
- 实现 RBAC 自动检查

**阶段 2：增强版本**
- 增加数据域自动过滤
- 增加脱敏自动应用
- 增加审计自动记录

**阶段 3：完整版本**
- 增加 Evidence 自动生成
- 增加超时重试机制
- 增加工具能力矩阵生成（文档自动化）

### 2.4 后端：Evidence 引擎

#### 2.4.1 设计目标

**GangQing 独有能力，所有数值结论必须可追溯到数据源。**

#### 2.4.2 核心能力

| 能力 | 说明 | 强制约束 |
|------|------|---------|
| **Citation** | 数据源引用（source/locator/time_range） | 数值结论必须绑定 |
| **Lineage** | 指标口径版本（metric/version/formula） | 计算结论必须绑定 |
| **ToolCallTrace** | 工具调用轨迹（toolName/duration/status） | 所有工具调用记录 |
| **增量更新** | SSE evidence.update 事件 | 实时输出 |
| **降级规则** | 缺失/不可验证/冲突时降级 | 必须输出 warning |

#### 2.4.3 Evidence 数据模型

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| evidenceId | string | ✅ | 证据唯一标识 |
| sourceSystem | enum | ✅ | ERP/MES/DCS/EAM/LIMS/Manual |
| sourceLocator | string | ✅ | 表名/接口名/文档路径 |
| timeRange | object | ✅ | start/end |
| toolCallId | string | ❌ | 关联的工具调用 ID |
| lineageVersion | string | ❌ | 指标口径版本 |
| dataQualityScore | float | ❌ | 数据质量评分（0-1） |
| confidence | enum | ✅ | Low/Medium/High |
| validation | enum | ✅ | verifiable/not_verifiable/out_of_bounds/mismatch |
| redactions | object | ❌ | 脱敏说明 |

#### 2.4.4 关键设计决策

**决策 1：如何判断 Evidence 是否可验证？**

| 条件 | validation 状态 | 说明 |
|------|----------------|------|
| 有数据源 + 时间范围 + 口径版本 | verifiable | 可追溯 |
| 缺少数据源或时间范围 | not_verifiable | 无法复核 |
| 数值越界或变化率异常 | out_of_bounds | 触发 guardrail |
| 多来源冲突且无法裁决 | mismatch | 数据不一致 |

**决策 2：如何处理 Evidence 缺失？**

| 场景 | 处理策略 | 用户体验 |
|------|---------|---------|
| 数值结论缺少 Evidence | 拒绝输出确定性数值 | 降级为"仅展示数据与来源" |
| 计算结论缺少 lineage_version | 拒绝输出或标注不确定 | 输出 warning 事件 |
| 工具调用无法生成 Evidence | 记录 tool.result(failure) | 返回结构化错误 |

**决策 3：如何优化 Evidence 存储与检索？**

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| 全部存 Postgres | 事务一致性 | 查询性能差 | ❌ |
| 全部存 Elasticsearch | 检索性能好 | 无事务保障 | ❌ |
| 双写（Postgres + ES） | 兼顾一致性与性能 | 复杂度高 | ✅ |

#### 2.4.5 实现策略

**阶段 1：基础版本**
- 实现 Evidence 数据模型
- 实现自动生成逻辑（从工具结果提取）
- 实现 SSE evidence.update 事件输出

**阶段 2：增强版本**
- 增加数据质量评分计算
- 增加置信度与验证状态判断
- 增加降级规则（缺失/不可验证/冲突）

**阶段 3：完整版本**
- 增加 Evidence 存储（Postgres + Elasticsearch）
- 增加 Evidence 检索接口
- 增加 Evidence 可视化（前端 Context Panel）

### 2.5 后端：SSE Envelope 统一封装

#### 2.5.1 设计目标

**统一 SSE 事件格式，强制字段（requestId/tenantId/projectId/sequence），便于审计与追溯。**

#### 2.5.2 核心能力

| 能力 | 说明 | 强制约束 |
|------|------|---------|
| **统一结构** | 所有事件同形（type + envelope + payload） | 不可绕过 |
| **强制字段** | requestId/tenantId/projectId/sequence | 缺失拒绝输出 |
| **序列号** | 单调递增，便于客户端检测丢包 | 自动生成 |
| **结构化错误** | error 事件同构 ErrorResponse | 便于解析 |
| **单行 JSON** | SSE 协议要求，便于解析 | 自动序列化 |

#### 2.5.3 SSE 事件类型

| 事件类型 | 用途 | 必填字段 | 可选字段 |
|---------|------|---------|---------|
| meta | 首事件，声明能力 | capabilities | - |
| progress | 阶段进度 | stage, message | stepId |
| tool.call | 工具调用开始 | toolCallId, toolName, argsSummary | - |
| tool.result | 工具调用结束 | toolCallId, toolName, status | resultSummary, error, evidenceRefs |
| message.delta | 回答增量 | delta | - |
| evidence.update | 证据链增量 | mode | evidence, evidenceId |
| warning | 降级/不确定项 | code, message | details |
| error | 结构化错误 | code, message, retryable, requestId | details |
| final | 结束事件 | status | summary |

#### 2.5.4 关键设计决策

**决策 1：为什么需要 Envelope 而非直接输出 payload？**

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| 直接输出 payload | 简洁 | 缺少上下文，难以审计 | ❌ |
| Envelope 封装 | 上下文完整，便于审计 | 冗余字段多 | ✅ |

**决策 2：如何处理 SSE 事件顺序？**

| 方案 | 说明 | 选择 |
|------|------|------|
| 无序 | 客户端自行排序 | ❌ 复杂度高 |
| 有序（sequence） | 服务端保证单调递增 | ✅ 简单可靠 |

**决策 3：如何处理 SSE 连接断开？**

| 场景 | 处理策略 | 客户端行为 |
|------|---------|-----------|
| 网络抖动 | 自动重连（指数退避） | 显示"重连中..." |
| 服务端主动断开 | 输出 final 事件后断开 | 显示"已完成" |
| 客户端主动取消 | 向下传播取消信号 | 停止推理/工具调用 |

#### 2.5.5 实现策略

**阶段 1：基础版本**
- 实现 SSEEnvelope 数据模型
- 实现 SSEStreamBuilder 封装类
- 实现基础事件输出（meta/progress/message.delta/final）

**阶段 2：增强版本**

### 2.6 后端：只读默认门禁

#### 2.6.1 设计目标

**GangQing 安全红线，写操作必须草案→审批→执行→回滚。**

#### 2.6.2 核心能力

| 能力 | 说明 | 强制约束 |
|------|------|---------|
| **意图识别** | QUERY/ANALYZE/ACTION_PREPARE/ACTION_EXECUTE | 不明确拒绝 |
| **写操作拦截** | 未满足门禁条件拒绝执行 | 返回 GUARDRAIL_BLOCKED |
| **草案生成** | 生成可编辑方案，不执行 | 允许 |
| **审批流程** | 按风险与域选择审批链 | 多签 |
| **受控执行** | Kill Switch + 白名单 + 幂等 | 可回滚 |
| **审计记录** | 所有阶段记录审计 | 不可篡改 |

#### 2.6.3 意图分类与策略

| 意图类型 | 说明 | 默认策略 | 门禁条件 |
|---------|------|---------|---------|
| QUERY | 只读查询 | 允许 | RBAC + 数据域 |
| ANALYZE | 分析与对比 | 允许 | RBAC + 数据域 |
| ALERT | 预警与推送 | 允许 | RBAC + 订阅权限 |
| ACTION_PREPARE | 草案生成 | 允许 | RBAC |
| ACTION_EXECUTE | 写操作执行 | 拒绝 | RBAC + 审批 + 白名单 + Kill Switch |

#### 2.6.4 写操作门禁条件

| 门禁 | 检查内容 | 不满足时 |
|------|---------|---------|
| **权限** | 用户是否有写权限（execution:execute:*） | 返回 FORBIDDEN |
| **审批** | 审批状态是否为 approved | 返回 GUARDRAIL_BLOCKED |
| **白名单** | 目标资源是否在允许范围 | 返回 GUARDRAIL_BLOCKED |
| **Kill Switch** | 熔断开关是否关闭 | 返回 GUARDRAIL_BLOCKED |
| **风险评估** | 高风险操作需升级审批链 | 返回 GUARDRAIL_BLOCKED |

#### 2.6.5 关键设计决策

**决策 1：如何识别写操作意图？**

| 方法 | 准确率 | 实现复杂度 | 选择 |
|------|--------|-----------|------|
| 关键词匹配 | 低（易绕过） | 低 | ❌ |
| 模型分类 | 中（需训练） | 中 | ✅ 阶段 1 |
| 工具调用分析 | 高（基于实际行为） | 高 | ✅ 阶段 2 |

**决策 2：如何处理模糊意图？**

| 场景 | 处理策略 | 用户体验 |
|------|---------|---------|
| 意图不明确 | 发起澄清对话 | 询问用户确认 |
| 倾向写操作但不确定 | 按只读处理 + 提示 | 安全优先 |
| 明确写操作 | 进入草案模式 | 生成可编辑方案 |

**决策 3：如何平衡安全与易用性？**

| 维度 | 安全优先策略 | 易用性优化 |
|------|-------------|-----------|
| 默认行为 | 只读 | 提供快捷审批通道（低风险操作） |
| 审批流程 | 多签 | 自动路由到合适审批人 |
| 执行控制 | Kill Switch + 白名单 | 白名单可配置化 |
| 回滚机制 | 强制创建回滚点 | 一键回滚 |

#### 2.6.6 实施策略

**阶段 1：意图识别（L1）**
- 实现基础意图分类（QUERY/ANALYZE/ACTION）
- 实现写操作拦截（返回 GUARDRAIL_BLOCKED）
- 实现审计记录

**阶段 2：草案生成（L4）**
- 实现草案生成逻辑（不执行）
- 实现草案编辑与提交
- 实现草案审计

**阶段 3：审批流程（L4）**
- 实现审批路由（按风险与域）
- 实现多签机制
- 实现审批状态机

**阶段 4：受控执行（L4）**
- 实现执行网关（Kill Switch + 白名单）
- 实现回滚机制
- 实现执行审计

### 8.3 关键决策记录（ADR）

#### ADR-001：不直接依赖 CopilotKit

**日期**：2026-02-26

**状态**：已接受

**背景**：
- CopilotKit 提供了优秀的 AI Copilot 架构参考
- GangQing 有工业场景特有的强制约束（只读默认、审计、证据链、多租户隔离）

**决策**：
不直接依赖 CopilotKit，借鉴其设计模式，自研核心组件。

**理由**：
1. 安全合规：CopilotKit 无法满足只读默认、审计不可篡改、Kill Switch 等强制约束
2. 私有化部署：模型与数据必须在内网，不可出厂
3. 证据链强制：所有数值结论必须可追溯（CopilotKit 无内置支持）
4. 多租户隔离：tenantId/projectId 强制过滤（CopilotKit 无内置支持）
5. 技术债务控制：避免 fork 并持续跟进上游更新

**后果**：
- 初期开发成本增加约 ¥226,000
- 长期维护成本降低约 ¥100,000/年
- 安全合规与定制化能力显著提升

---

#### ADR-002：使用装饰器而非基类继承

**日期**：2026-02-26

**状态**：已接受

**背景**：
- 需要简化工具注册，自动化 RBAC/脱敏/审计
- 可选方案：基类继承 vs 装饰器

**决策**：
使用装饰器模式实现工具注册。

**理由**：
1. 声明式：代码简洁，易于理解
2. 灵活性：不强制继承，可与现有代码兼容
3. 可组合：可叠加多个装饰器
4. 借鉴 CopilotKit：与 `useCopilotAction` 理念一致

**后果**：
- 运行时检查（而非编译时）
- 类型提示较弱（需额外文档）
- 提供低级 API 作为备选方案

---

#### ADR-003：使用依赖注入而非中间件

**日期**：2026-02-26

**状态**：已接受

**背景**：
- 需要自动提取并校验请求上下文（requestId/tenantId/projectId）
- 可选方案：中间件 vs 依赖注入

**决策**：
使用 FastAPI Depends 实现依赖注入。

**理由**：
1. 显式声明：每个路由明确声明依赖
2. 易于测试：可 mock Depends
3. 类型安全：FastAPI 自动类型检查
4. 借鉴 CopilotKit：与上下文管理理念一致

**后果**：
- 每个路由需声明 `ctx: RequestContext = Depends(get_request_context)`
- 代码略显冗余（但更清晰）

---

#### ADR-004：双写审计日志（Postgres + Elasticsearch）

**日期**：2026-02-26

**状态**：已接受

**背景**：
- 审计日志需要事务一致性（append-only）
- 审计日志需要高效检索（按 requestId/userId/时间范围）

**决策**：
双写审计日志到 Postgres（主存证）和 Elasticsearch（检索增强）。

**理由**：
1. 事务一致性：Postgres 保证 append-only
2. 检索性能：Elasticsearch 支持复杂查询
3. 容灾：双写提供冗余

**后果**：
- 实现复杂度增加
- 需要保证双写一致性（异步写 ES，失败重试）
- 存储成本增加

#### ADR-005：SSE 优先，WebSocket 作为备选

**日期**：2026-02-26

**状态**：已接受

**背景**：
- 需要流式输出对话与证据链
- 可选方案：SSE vs WebSocket

**决策**：
SSE 作为主协议，WebSocket 作为备选（支持双向通信场景）。

**理由**：
1. 简单性：SSE 更简单，易于实现与调试
2. 兼容性：SSE 易于穿透企业网络与网关
3. 自动重连：浏览器原生支持
4. 借鉴 CopilotKit：CopilotKit 也优先使用 SSE

**后果**：
- 单向通信（服务端 → 客户端）
- 取消需要额外机制（客户端断开 → 服务端检测）
- WebSocket 作为备选（需要双向通信时）

### 8.4 FAQ

#### Q1：为什么不直接使用 CopilotKit？

**A**：CopilotKit 无法满足 GangQing 工业场景的强制约束：
- 只读默认（写操作必须审批）
- 审计不可篡改（append-only）
- 证据链强制（所有数值可追溯）
- 多租户隔离（tenantId/projectId 强制过滤）
- Kill Switch（全局熔断）
- OT 写入门禁（专用通道 + 二次确认）

自研方案虽然初期成本高，但长期成本更低，且安全合规与定制化更优。

---

#### Q2：如何保证自研组件的质量？

**A**：通过以下措施保证质量：
1. **借鉴成熟模式**：参考 CopilotKit 的优秀设计模式
2. **强制测试覆盖**：单元测试 > 80%，冒烟测试覆盖核心链路
3. **契约测试**：自动化测试覆盖所有对外接口
4. **代码审查**：所有代码必须经过审查
5. **渐进式演进**：分阶段实施，每个阶段验收

---

#### Q3：如何平衡自动化与灵活性？

**A**：通过分层策略平衡：
1. **标准工具**：100% 自动化（装饰器 + RBAC + 脱敏 + 审计）
2. **特殊工具**：提供钩子函数（before/after）
3. **复杂工具**：提供低级 API（手动控制）

开发者可根据场景选择合适的抽象层级。

---

#### Q4：如何处理性能问题？

**A**：通过以下策略优化性能：
1. **异步化**：Evidence 生成异步化
2. **批处理**：SSE 事件批处理
3. **缓存**：工具调用结果缓存
4. **并发控制**：模型推理并发控制
5. **降级**：超时/过载时降级

目标：简单查询 P95 < 2 秒，复杂分析 P95 < 30 秒。

---

#### Q5：如何保证多租户隔离？

**A**：通过以下机制保证隔离：
1. **强制校验**：RequestContext 强制校验 tenantId/projectId
2. **自动过滤**：所有数据访问自动叠加过滤
3. **跨域拒绝**：检测到跨域访问立即拒绝并审计
4. **自动化测试**：测试覆盖跨域访问场景

---

#### Q6：如何处理 Evidence 缺失？

**A**：通过降级规则处理：
1. **数值结论缺少 Evidence**：拒绝输出确定性数值，降级为"仅展示数据与来源"
2. **计算结论缺少 lineage_version**：输出 warning 事件，标注不确定
3. **工具调用无法生成 Evidence**：记录 tool.result(failure)，返回结构化错误

核心原则：宁可降级，不可编造。

---

#### Q7：如何保证审计日志不可篡改？

**A**：通过以下机制保证：
1. **append-only**：Postgres 表设计为 append-only
2. **权限控制**：只有审计员可读，禁止修改
3. **双写**：Postgres + Elasticsearch 双写，提供冗余
4. **监控**：监控审计日志写入失败

---

#### Q8：如何处理 Kill Switch？

**A**：Kill Switch 实现策略：
1. **全局开关**：配置外部化（环境变量或配置中心）
2. **实时生效**：开关变更立即生效（无需重启）
3. **审计记录**：开关启停记录审计
4. **可观测**：开关状态可查询
5. **告警**：开关启用时触发告警

---

#### Q9：如何处理 OT 写入？

**A**：OT 写入门禁策略：
1. **禁止直连**：AI 不可直连 DCS/PLC
2. **专用通道**：通过执行网关 + 专用通道
3. **二次确认**：OT 侧人员必须确认
4. **全程审计**：所有操作记录审计
5. **可回滚**：支持一键回滚
