# Project Rules (AGENTS) - GangQing

## 0) 绝对红线 (Absolute Directives)
所有 AI 助手在执行任务时，必须将以下原则置于最高优先级：
1. **中文沟通**：始终使用中文回复与编写文档（代码注释、英文错误码除外）。
2. **防过度设计 (KISS & YAGNI)**：
   - **严格禁止**超前实现、预测性编码、炫技式抽象（如无谓的接口/基类/泛型体操）。
   - **只实现当前任务（T#）明确要求的功能**。多写一行无用代码视为 Blocker。
   - 首选项目已有基础库 / 标准库，拒绝重复造轮子。
3. **真实集成 (No Fake Mocks)**：
   - **所有代码必须使用真实服务进行集成和测试**。
   - **禁止使用 mock**：集成/冒烟测试中不得使用 mock 对象、mock 服务。
   - **配置缺失 => 测试失败**：缺少配置必须抛出清晰英文错误，禁止使用 `test.skip()`。
4. **零技术债务 (Zero Tech Debt)**：
   - **零遗留项原则**：任务完成时不得存在任何 `TODO`、`待补充`、`后续优化`。
   - **审查问题闭环**：所有审查报告中的问题（Blocker/Critical/Major/Minor）必须 100% 修复或证明不适用。

## 1) 编码与架构规范

### 1.1 全栈语言与校验边界
- **前端 (TS)**：强制 TypeScript Strict（**禁止 `any`**）。输入/输出使用 **Zod** 校验。
- **后端 (Python)**：强制类型注解。所有 API request/response、工具参数使用 **Pydantic** 模型。
- **错误处理**：抛出的异常必须结构化（code, message, requestId, retryable），`message` 必须使用**英文**。

### 1.2 配置化开发 (强制)
- **所有配置外部化**：URL、端口、超时等必须通过环境变量或配置文件管理，禁止硬编码。
- **配置加载与校验**：必须使用 Zod/Pydantic 在加载时校验。
- **本地开发 `.env.local`**：读取优先级：**进程环境变量** > **根目录 `.env.local`**。`.env.example` 必须完整。

### 1.3 GangQing 核心架构约束
- **只读默认**：未授权前不得执行写操作。写操作必须走“草案 → 审批 → 执行 → 审计”链路。
- **证据链 (Evidence)**：数值结论必须能映射到数据源、时间范围、口径版本。
- **审计覆盖**：必须覆盖查询、工具调用（脱敏）、审批、写操作。

## 2) 文档生成与任务状态

### 2.1 文档落盘规则
- 必须以 **Markdown (.md)** 格式在实际目录中创建/更新文件，**绝不能只在聊天窗口输出**。
- 若写入失败（如被中断），必须在聊天中用 ` ```markdown ` 代码块输出完整内容供用户复制。

### 2.2 任务提示词联动 (强制)
- 当更新非日志类文档（如 API 规范、设计方案）时，**必须同步更新后续未开始任务的 Prompt**（`docs/task-prompts/*.md`）：
  - 在 Prompt 中追加“权威参考文档”引用。
  - 勾选该任务 Prompt Checklist 中的“Doc References Updated”。

### 2.3 任务完成标记
- 只有在**测试全部通过且生成验收日志后**，才能在 `docs/tasks.md` 中打钩 `[x]` 并标记完成。

## 3) 自动化验收与测试规范

### 3.1 测试双覆盖要求 (必须运行并通过)
每个任务必须同时具备以下两类测试，并在验收日志中记录执行结果：
1. **单元测试 (Unit Test)**：覆盖纯逻辑、边界、结构化错误。小改动 ≥ 2，新增 API ≥ 6，跨模块 ≥ 10。
2. **冒烟测试 (Smoke Test)**：覆盖真实端到端链路。小改动 ≥ 1，新增 API ≥ 2，跨模块 ≥ 3。

### 3.2 验证命令执行标准
- **Python 虚拟环境**：强制使用根目录 `.venv`。命令必须在激活环境或使用 `.venv/bin/python` 下执行。
- **后端校验**：
  - 最低编译校验：`python -m compileall backend`
  - 单测执行：`pytest -q`
  - 冒烟测试：运行 `backend/scripts/*_smoke_test.py` 或指定脚本。
- **前端校验**：在 `web/` 目录执行 `npm run build`。
- *注：若因环境缺失无法运行，必须在验收日志中记录原因与补测计划，不得直接标记完成。*

## 4) 审查与验收交付物

### 4.1 报告命名规范
- **验收日志**：`reports/YYYY-MM-DD_T{N}_short-slug.md`
- **审查报告**：`reports/YYYY-MM-DD_T{N}_short-slug-review.md`

### 4.2 审查阻断红线 (Review Blockers)
审查 AI 发现以下问题将直接阻断验收，开发 AI 必须无条件返工：
| 违规类型 | 表现特征 | 强制动作 |
|---------|---------|---------|
| **过度设计** | 包含当前需求外未使用的字段、API、多余的工厂模式/基类 | 立即删除冗余代码 |
| **契约漂移** | 接口实现与 `docs/contracts/` 定义的 Schema 不一致 | 对齐权威设计文档 |
| **未校验配置** | `os.getenv` 后直接使用，没有异常捕获和类型转化 | 引入 Pydantic/Zod 校验 |
| **死代码** | 存在未使用的变量、私有函数、被大段注释掉的历史代码 | 立即清理 |
| **测试造假** | 验收日志说测试通过，但使用了 `mock` 替代真实数据库 | 改为真实服务冒烟测试 |

### 4.3 零遗留项声明
验收日志必须包含**【零遗留项声明】**章节，明确列出：
- [ ] 所有审查问题已修复（或附不适用原因及新任务 ID）。
- [ ] 文档完整性确认（API/错误码/配置项已完整交付）。
- [ ] 测试覆盖确认（单元/冒烟/回滚测试已通过）。
- [ ] 没有任何 TODO、待补充、后续优化等遗留项。
